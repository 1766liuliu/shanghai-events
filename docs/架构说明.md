# 架构说明

## 三层解耦

```
采集层(本机 Python)  →  数据+服务层(微信云开发)  →  前端(微信小程序)
   抓取/清洗/打标            云数据库 + 云函数            列表/详情/收藏/提醒
```

## 数据流

1. **采集层**（`collector/`，跑在本机）：从各数据源抓取 → 清洗去重 → 亲子/适龄/价格打标 → 写入。
   - 调试期写本地 `data/events.json`（`store/local_json.py`）。
   - 正式写微信云数据库 `events` 集合（`store/cloudbase.py`）。
2. **云后端**（`cloud/`）：
   - `getEvents` 给小程序读列表/详情；
   - `weeklyDigest` 定时生成本周精选；
   - `ticketWatch` 定时检查开票时间并发订阅消息。
3. **小程序**（`miniprogram/`）：家人浏览、筛选、收藏、订阅开票提醒。

## 为什么采集层与小程序解耦

- 商业票务平台风控会快速封云函数固定出口 IP，本机/可控节点更稳；
- 把法律灰色的抓取与“干净”的小程序前端物理隔开，降低小程序侧合规暴露；
- 契合“工具默认本机运行”的习惯。

## 采集节点位置

- **现阶段**：本机（用户的 Mac），用于开发调试。
- **注意**：海外抓国内政府站点可能慢/受限，调试够用。
- **长期**：稳定后迁到上海常驻小主机。

## 关键约束

- 个人号小程序：类目受限、订阅消息仅「一次性」。MVP 以工具类自用、不公开；扩展转企业主体。
- 仅自用、不二次分发、不商用。

# 上海家庭活动雷达 — 产品需求与技术方案（PRD v0.1）

> 状态：待批准 / 设计调试阶段
> 创建：2026-06-14
> 形态：微信小程序（家人各自浏览订阅）
> 说明：本文件是"动手写代码前讲清楚做什么、给谁用、有哪些功能、先做哪个"的设计蓝图（PRD = 产品需求文档）。设计调试稳定后再决定永久存放目录。

---

## 0. 一句话目标

让在上海的家人提前发现、筛选、订阅有价值的**体育赛事 / 行业展会 / 文化演出**，**亲子内容优先**，并能在开票时被提醒；爸爸在海外每周收到一份"上海本周精选"。

定位：**信息发现 + 提醒**，不卖票、不代抢，点击跳转官方购票/报名渠道。

## 2. 用户与场景

**用户**：用户本人（海外）+ 上海家人（妈妈、孩子），各自微信打开。

**核心场景**
1. 妈妈周末前打开 → 看"本周上海亲子可去"清单 → 收藏两个。
2. 孩子想看的某演出 → 加"关注" → 开票当天收到提醒 → 跳官方渠道购票。
3. 爸爸在海外 → 每周一收到"上海本周精选"推送 → 转发家庭群 / 并入周末行程。
4. 找展会 → 筛选"国家会展中心 + 本月 + 适合带娃" → 看详情。

## 4. 技术架构（解耦三层）

```
┌─────────────────────────────────────────────┐
│  采集层  (本机 Mac, Python — 调试期)            │
│  · 稳定源抓取(会展平台/体育局/久事/文化云/本地宝) │
│  · 商业票务抓取(大麦/猫眼/票星球, 低频+重试+兜底) │
│  · 清洗去重 + 规则/AI 打标(亲子/适龄/区域/价格)  │
│  · 监控"开票状态"变化                          │
└───────────────┬─────────────────────────────┘
                │ 写入(云开发 OpenAPI / HTTP)
                ▼
┌─────────────────────────────────────────────┐
│  数据+服务层  (微信云开发 CloudBase)            │
│  · 云数据库: events / watch / users / favs     │
│  · 云函数: 读接口、订阅管理、推送触发、周报生成    │
│  · 定时触发器: 轻量任务(周报、开票轮询)          │
└───────────────┬─────────────────────────────┘
                │ 小程序 SDK(只读+订阅)
                ▼
┌─────────────────────────────────────────────┐
│  前端  (微信小程序)                            │
│  列表/筛选 · 详情 · 收藏 · 关注 · 订阅 · 周报    │
└─────────────────────────────────────────────┘
```

**为什么采集层与小程序解耦**
1. 商业平台风控会快速封云函数固定出口 IP，本地节点 + 可控代理更稳；
2. 契合"自建工具默认本地运行"的习惯；
3. 把法律灰色的抓取与"干净"的小程序前端物理隔开，降低小程序侧合规暴露。

**调试期采集放本机的代价**：用户在海外，抓上海/国内政府站点可能慢或对境外 IP 不友好。调试够用；稳定后把采集迁到上海常驻小主机（长期布局）。

## 6. 数据模型（云数据库集合）

- **events**：`title, type(体育/展会/演出), venue, district, startDate, endDate, openTicketTime, priceRange, kidFriendly(bool), ageRange, tags[], officialUrl, source, status, cover`
- **watch**：用户关注的活动（驱动开票提醒）`openid, eventId, notifyOnSale`
- **favorites**：`openid, eventId`
- **users**：`openid, interests[], pushOptIn`
- **weekly**：每周精选快照

---

## 7. 提醒机制（已规避个人号限制）

- **机制**：微信小程序**一次性订阅消息** —— 用户在活动页点"开票提醒我"即授权一次推送；采集层监控到该活动 `openTicketTime` 到达或状态变"在售" → 云函数触发推送。
- **约束**：个人号拿不到"长期订阅"，所以是"一次一授权"；周报同理需用户先点订阅。这是个人号阶段的固有体验损耗，转企业主体后可优化。

---

## 8. 亲子/价值过滤逻辑（核心差异化）

采集层执行（规则 + 可选 AI 兜底）：
- `kidFriendly`：关键词（亲子/儿童/科普/展/演）+ 场馆类型 + 价格带 + 时段（周末/白天）
- `ageRange`：从文案抽取（如"3–8 岁"）
- 周报排序：亲子优先 → 区域就近 → 价格友好 → 临近开票

---

## 9. 里程碑与工作量（单人 AI 辅助估算）

| 阶段 | 内容 | 工期 |
|---|---|---|
| M1 采集层 | 3–4 稳定源抓取 + 清洗 + 打标 + 写云库 | 1–2 周 |
| M2 云后端 | 数据模型 + 读接口 + 周报云函数 | 3–5 天 |
| M3 小程序前端 | 列表/筛选/详情/收藏 | 1–2 周 |
| M4 提醒 | 一次性订阅 + 开票监控 | 2–3 天 |
| M5 商业源 + 联调 | 大麦等接入 + 兜底 + 测试 | 3–5 天 |
| **MVP（M1–M3）** | 可用浏览版 | **约 3 周** |
| **完整（+M4/M5）** | 含提醒 + 商业源 | **约 5 周** |

兼职推进则翻倍。

## 10. 风险登记

| 风险 | 影响 | 对策 |
|---|---|---|
| 商业票务抓取失效/封禁 | 抢票提醒断 | 低频+代理+人工兜底关注列表；官方公告为主源 |
| 个人号类目审核不过 | 无法上架 | MVP 以工具类自用、不公开；扩展转企业主体 |
| 采集脚本腐烂（用户在海外） | 数据过期 | 锚定官方稳定源；失败告警推微信；商业源可降级关停 |
| 本机采集节点海外抓国内站慢/受限 | 调试体验差 | 调试期可接受；稳定后迁上海小主机 |
| 长期订阅消息不可用 | 提醒需反复授权 | 一次性订阅；转企业主体后优化 |
| 法律/条款（灰色抓取） | 合规暴露 | 仅自用、不二次分发、不商用；前端不展示抓取来源水印 |

## 11. 成本

- 微信小程序注册（个人号）：免费
- 微信云开发：有免费额度，自用量级基本够，超出按量（预计每月几元到几十元）
- 本机采集节点：用现有 Mac = 0；将来小主机约几百元一次性
- 代理 IP（商业源需要时）：按需，每月几十元
